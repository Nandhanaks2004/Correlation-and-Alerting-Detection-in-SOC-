Phase 1

Why Single Logs Are Not Enough

Imagine you are a SOC analyst. You receive one event. Can you determine if it is an attack? Let us find out.

Scenario: You See These Events — One at a Time

📅 Event 1 — Failed Login at 10:01 AM
User: John | Target: DC01 | Result: Failure
Is this an attack? Not enough evidence. A single failed login is usually a typo or forgotten password. Alone, it means almost nothing.
📅 Event 2 — Successful Login at 10:05 AM
User: John | Target: DC01 | Result: Success
Is this an attack? Still not clear. John successfully logged in after a failed attempt. This could simply be John trying again with the correct password.
📅 Event 3 — File Opened at 10:07 AM
User: John | File: Budget.xlsx | Action: Read
Is this an attack? Not necessarily. John opened a spreadsheet. Employees open files every day. Alone, this event is completely benign.
⚖ Now Let Us Look at the Full Picture

The Complete Attack Timeline

10:02 — Failed Login
↓
10:03 — Failed Login
↓
10:04 — Failed Login
↓
10:05 — Failed Login
↓
10:06 — Successful Login
↓
10:07 — Sensitive File Accessed
↓
10:08 — Data Downloaded

Now Is It An Attack?

Yes. What changed? A pattern emerged. Individually, each event could be innocent. But together — failed logins followed by a successful login, sensitive file access, and data download — they tell a clear story of a breach.

Key Takeaway

Individual events tell stories. Patterns reveal attacks. This single concept is the foundation of all SOC detection methodologies.

😈 Attacker Perspective

Attackers rely on their actions appearing normal individually. They know that one failed login is ignored, one file access is not suspicious. They hope each event hides in the noise.

👥 SOC Analyst Perspective

SOC analysts know that individual events are noise. The skill is recognizing patterns — connecting the dots between seemingly unrelated events to reveal the attack hiding in plain sight.

Phase 3

How Detection Rules Work

A SIEM does not magically know something is malicious. Detection rules define what to look for. Without them, the SIEM is just a storage system.

Condition — What activity are we looking for?
+
Threshold — How many events?
+
Time Window — Within what period?
=
Detection Rule → Alert

Condition

What specific activity triggers the rule? Failed logins, successful logins from new countries, PowerShell execution, USB insertion.

EventCode=4625

Threshold

How many times must the condition occur? 1 failed login is normal. 10 in 5 minutes is suspicious. The threshold separates noise from signal.

Count > 10

Time Window

Within what time period must the threshold be crossed? 10 failed logins in 24 hours is different from 10 in 2 minutes. The time window determines urgency.

Within 5 Minutes

Complete Rule Example

IF EventCode=4625
AND Count > 10
WITHIN 5 Minutes

THEN Generate "Brute Force" Alert
Phase 4

Alert Generation

When a detection rule's conditions are met, an alert is generated. But understanding what an alert actually means is one of the most important lessons for new analysts.

ALERT ≠ ATTACK

An alert means something requires investigation. It does not mean an attack is occurring. This distinction is critical for every SOC analyst.

Example Alert Notification

Alert Name: Possible Brute Force Attack
Severity: Medium
User: Administrator
Source IP: 192.168.1.100
Failed Attempts: 25
Time Window: 3 Minutes

The alert tells the analyst what happened, who was involved, and how critical it might be. The analyst must now investigate to determine if this is a real attack or a false positive.

Alert Lifecycle

Logs Generated ↓ Rule Triggered ↓ Alert Created ↓ Analyst Reviews ↓ Investigation ↓ Decision
Phase 5

Alert Severity Levels

Not all alerts are equal. SOC teams prioritize using severity levels. Understanding severity helps analysts know what needs immediate attention.

Critical
Examples: Credential dumping (Mimikatz), ransomware execution, domain admin compromise, active malware outbreak.
Analyst action: Immediate response. Escalate to incident response team. Contain the threat.
High
Examples: Suspicious admin activity, unexpected privileged access, malware detection on critical server.
Analyst action: Investigate within minutes. If confirmed, escalate to Tier 2/3.
Medium
Examples: Repeated failed logins, unusual time access, blocked outbound connection.
Analyst action: Review within the shift. Look for patterns. Determine if context suggests escalation.
Low
Examples: Single failed login, single blocked port scan, informational event.
Analyst action: Review during low-priority time. Usually closed as benign after quick check.
Informational
Examples: User logged in, service started, system rebooted.
Analyst action: No action required. Used for audit and historical reference.

SOC Rule of Thumb: Higher severity = faster response. Critical alerts may require action within minutes. Informational alerts may never require action.

Phase 6

False Positives vs True Positives

One of the most important topics for new SOC analysts. Every alert falls into one of two categories after investigation.

False Positive

Alert Generated — No Attack Occurred

The detection rule triggered, but the activity was benign. The alert was a "false alarm."

Example: User Sarah forgot her password. She attempted login 5 times, triggering the "Brute Force" rule. Investigation shows it was Sarah herself, not an attacker. The alert is closed as a false positive.
True Positive

Alert Generated — Attack Confirmed

The detection rule triggered correctly. Malicious activity was occurring.

Example: 100 failed logins for Administrator from IP 203.0.113.50 in 2 minutes. Investigation confirms the IP is known for brute-force attacks. The alert is a true positive and requires incident response.
Alert Generated
↓
Analyst Investigation
↓
False Positive True Positive

Enterprise Context

In a typical SOC, 70-90% of alerts are false positives. A significant part of a Detection Engineer's job is tuning rules to reduce false positives while maintaining detection of real threats. Every false positive wastes analyst time and attention.

Phase 7

Detection Engineering Fundamentals

Alerts do not appear by magic. Someone has to create them. That someone is a Detection Engineer — and understanding their role helps you appreciate how SOC detections are built.

Detection Engineer

A specialist who creates, maintains, and improves detection rules to identify malicious activity while minimizing false positives.

Responsibilities

  • • Create new detection rules
  • • Tune existing rules to reduce false positives
  • • Research new attack techniques
  • • Map detections to MITRE ATT&CK framework
  • • Collaborate with SOC analysts on gaps

Example Detection Rule

# Brute Force Detection Rule
IF EventCode=4625
AND Count > 10
WITHIN 5 Minutes
THEN Alert("Possible Brute Force")

Alerts are not magic. Rules create alerts. Understanding this distinction is what separates analysts who just investigate from analysts who can improve the detection pipeline.

One event alone? Maybe suspicious, maybe not.

Why Correlation Matters Here

A single phishing email is not evidence of a breach. A single PowerShell execution could be legitimate admin work. A single network connection could be a user browsing the web.

But when a correlation rule links email delivery + attachment opened + PowerShell + unknown external connection + malware download, the confidence level becomes extremely high. The pattern is unmistakable.

Correlation transforms uncertainty into confidence.

Knowledge Check

Test Your Understanding

These questions verify you understand how correlation, detection rules, and alerts work in a SOC environment.

1. Why can a single log rarely prove an attack?

Correct! A single failed login could be a typo. A single file access could be normal work. But when multiple events form a pattern — failed logins followed by success, privilege escalation, data access — the attack becomes visible.

2. What is correlation in cybersecurity?

Correct! Correlation is the process of connecting multiple seemingly unrelated events to reveal a security pattern. It transforms individual data points into actionable intelligence.

3. What is an alert in a SIEM?

Correct! An alert is a notification that a detection rule's conditions have been met. It does not mean an attack is happening — it means something requires investigation.

4. What is the difference between an Alert and an Incident?

Correct! Alert ≠ Incident. An alert becomes an incident only after investigation confirms malicious activity. Many alerts are false positives that never become incidents.

5. What is a False Positive?

Correct! A false positive is an alert that fires on benign activity. Example: a user mistyping their password triggers a "Brute Force" alert, but investigation shows it was the legitimate user.

6. What is a True Positive?

Correct! A true positive means the detection rule correctly identified malicious activity. The analyst investigated and confirmed an actual attack or policy violation.

7. Who creates and maintains detection rules in a SOC?

Correct! Detection Engineers specialize in creating, testing, and tuning detection rules. They work with SOC analysts to identify gaps and reduce false positives.

8. Why are correlation rules important in a SIEM?

Correct! Correlation rules are what transform a SIEM from a log storage system into a detection platform. They identify patterns across multiple data sources that indicate malicious activity.
